# ADR 002: Separate PasswordEncoder Configuration

## Status
Accepted

---

## Context

В системе реализуется модуль аутентификации (`AuthService`), который использует `PasswordEncoder` для безопасного хэширования паролей пользователей при регистрации и проверки пароля при логине.

На текущем этапе проекта Spring Security ещё не подключён полностью (`SecurityConfig` будет добавлен в последующих модулях). Однако компонент `PasswordEncoder` уже необходим на уровне application layer для реализации бизнес-логики аутентификации.

Возник вопрос архитектурного характера: где корректно объявить бин `PasswordEncoder`.

---

## Decision

Принято решение выделить отдельный инфраструктурный конфигурационный класс `PasswordEncoderConfig` и зарегистрировать в нём бин `BCryptPasswordEncoder`.

---

## Rationale

### 1. Избежание временной архитектурной зависимости

`SecurityConfig` будет добавлен позже (в рамках Security модуля). Привязка `PasswordEncoder` к нему на текущем этапе создаёт временную зависимость, которая потребует рефакторинга `AuthService` в будущем.

---

### 2. Разделение ответственности (SRP)

* `PasswordEncoderConfig` отвечает исключительно за криптографические компоненты.
* `SecurityConfig` будет отвечать за HTTP security, фильтры, JWT и авторизацию.

---

### 3. Снижение связности (Loose Coupling)

Выделение отдельного конфигурационного класса позволяет:

* избежать циклических зависимостей;
* не связывать бизнес-логику с security-инфраструктурой;
* сохранить стабильный контракт `PasswordEncoder` для `AuthService`.

---

### 4. Готовность к расширению

В будущем возможны изменения криптографического алгоритма:

* `Argon2PasswordEncoder`
* `DelegatingPasswordEncoder`
* миграция существующих хэшей

Отдельный конфигурационный класс упрощает замену реализации без изменения `AuthService`.

---

## Consequences

### Positive

* Чёткое разделение конфигурационных обязанностей
* Упрощение будущего `SecurityConfig`
* Снижение риска рефакторинга `AuthService`
* Явное выделение инфраструктурного слоя

---

### Negative

* Дополнительный конфигурационный класс
* Распределённая конфигурация security-компонентов требует дисциплины в архитектуре

---

## Notes

Подход соответствует следующим архитектурным принципам:

* Dependency Inversion Principle (DIP)
* Separation of Concerns (SoC)
* Hexagonal Architecture (изоляция инфраструктуры от application logic)

---

## Decision Summary

`PasswordEncoder` выделяется в отдельный инфраструктурный конфигурационный класс (`PasswordEncoderConfig`) для предотвращения преждевременной зависимости от будущего `SecurityConfig` и для сохранения чистой и расширяемой архитектуры `AuthService`.
