# 001 - Hexagonal Architecture

## Контекст

Проект представляет собой backend-приложение на Spring Boot для управления турами, заказами и пользователями.

Система будет развиваться, включая:
- бизнес-логику
- работу с базой данных
- REST API
- потенциальные внешние интеграции

Возникла необходимость выбрать архитектурный подход, который обеспечит:
- масштабируемость
- тестируемость
- изоляцию бизнес-логики от инфраструктуры

---

## Решение

Принято решение использовать **Hexagonal Architecture (Ports & Adapters)**.

Основные принципы:

- бизнес-логика выделяется в отдельный domain слой
- взаимодействие с внешним миром происходит через порты (interfaces)
- инфраструктура реализует адаптеры для этих портов
- зависимости направлены внутрь (towards domain)

Структура проекта разделена на:

- domain — ядро системы
- application — use case слой
- infrastructure — адаптеры (web, persistence)
- config — конфигурация Spring

---

## Последствия

### Положительные:

- бизнес-логика изолирована от Spring и БД
- высокая тестируемость domain слоя
- легко заменять инфраструктуру (DB, API, UI)
- строгие границы ответственности между слоями

---

### Отрицательные:

- увеличивается количество кода (interfaces + mappers)
- сложнее стартовая разработка по сравнению с layered architecture
- требуется дисциплина соблюдения зависимостей

---

### Компромиссы:

- дополнительная сложность ради долгосрочной поддерживаемости
- принятие более строгой архитектурной модели вместо простого MVC подхода
